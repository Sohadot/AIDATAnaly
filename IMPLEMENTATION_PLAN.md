## 1. Purpose of This Plan

This document defines the governed implementation plan for building AIDAtanaly.com from ratified doctrine into a public reference system.

AIDAtanaly is not being implemented as a generic website.

It is being implemented as a governed Category Intelligence Factory.

The governing thesis remains:

> Stages are states. Value lives in transitions.

Implementation must convert that thesis into:

- stable public routes,
- governed reference pages,
- machine-readable data files,
- scanner-ready diagnostic logic,
- reusable interface components,
- internal link integrity,
- SEO-safe structure,
- performance-safe delivery,
- accessibility-safe design,
- monetization readiness,
- and future buyer-facing credibility.

## 2. Implementation Principle

AIDAtanaly implementation follows this rule:

> Governance first. Data second. Components third. Pages fourth. Scanner fifth. Release last.

No public indexed reference release may occur before the Required Launch route set is complete.

The first public indexed reference release must include the full 41-route Required Launch set defined in `ROUTE_MAP.md`.

Smaller builds may exist only as:

- private preview,
- internal review,
- staging environment,
- soft launch with `noindex`,
- or controlled non-indexed demonstration.

No incomplete indexed release may be treated as the official AIDAtanaly public reference launch.

## 3. Technical Posture

AIDAtanaly must use a static-first implementation model.

**Required Technical Constraints**

- Static-first architecture.
- Semantic HTML.
- CSS variables.
- Minimal JavaScript.
- Vanilla JavaScript for scanner functionality where needed.
- Reference pages readable without JavaScript.
- No external JavaScript dependencies by default.
- No analytics before a governance decision.
- No WebGL or 3D unless separately governed.
- No decorative motion.
- No generated route outside `ROUTE_MAP.md`.
- No page outside `PAGE_BLUEPRINTS.md`.
- No scanner logic outside `SCANNER_MODEL.md`.
- No JSON definition contradicting ratified Markdown standards.

**Preferred Stack**

```
Static HTML
CSS variables
Vanilla JavaScript
JSON data files under /data/
Automated validation scripts
Generated sitemap.xml
robots.txt governance
GitHub Pages or equivalent static hosting
```

Any framework may be used only if it does not weaken static readability, SEO, accessibility, performance, governance traceability, or build simplicity.

## 6. Data Implementation Layer

The first implementation must create five machine-readable data files:

```

/data/ati-vectors.json

```

These files must match the ratified Markdown standards.

---

## 7. Data File Requirements

### 7.1 `/data/ati-standard.json`

Must include:

- ATI version,
- D1–D7 dimensions,
- dimension weights,
- diagnostic thresholds,
- Evidence Confidence levels,
- vector weighting,
- E0 rule,
- Partial Profile rule reference,
- canonical route references.

Required checks:

- D1–D7 weights total 100%.
- Evidence Confidence is not included in score.
- Diagnostic thresholds match `ATI_STANDARD.md`.

---

### 7.2 `/data/ati-vectors.json`

Must include:

- `ati.vector.t1`
- `ati.vector.t2`
- `ati.vector.t3`
- `ati.vector.t4`

Each vector must include:

- vector ID,
- transition,
- transition name,
- core diagnostic question,
- strong signals,
- weak signals,
- canonical route,
- associated failure mode IDs.

---

### 7.3 `/data/tfo-failure-modes.json`

Must include all 21 TFO failure modes and the Measurement Gap diagnostic constraint.

Each object must include:

- stable ID,
- name,
- primary vector,
- transition,
- diagnostic class,
- canonical route,
- definition,
- symptoms,
- detection signals,
- scoring impact,
- intervention layers,
- related failure modes.

No failure mode ID may exist in JSON unless it exists in `TFO_ONTOLOGY.md`.

---

### 7.4 `/data/intervention-layers.json`

Must include only the ten approved intervention layers:

- Audience Intervention
- Message Intervention
- Offer Intervention
- Proof Intervention
- UX Intervention
- Pricing Intervention
- Timing Intervention
- Attribution Intervention
- Lifecycle Intervention
- Sales-Assist Intervention

It must preserve contextual mappings:

```
Channel Intervention → Audience Intervention or Message Intervention
Measurement Governance → Attribution Intervention
```

These contextual labels are not standalone intervention layers.

---

### 7.5 `/data/scanner-model.json`

Must include:

- scanner version,
- answer scale,
- question IDs,
- vector mappings,
- dimension mappings,
- Evidence Confidence rules,
- failure mode mappings,
- intervention mappings,
- output templates,
- diagnostic thresholds,
- canonical reference links,
- Partial Profile rule.

It must match `SCANNER_MODEL.md`.

## 10. JavaScript Requirements

JavaScript is permitted for scanner interactivity.

Reference pages must remain readable without JavaScript.

Scanner JavaScript must:

- load or mirror the scanner model,
- collect structured answers,
- calculate vector scores,
- assign Evidence Confidence,
- apply E0 and Partial Profile rules,
- assign failure modes,
- map intervention layers,
- produce controlled output language,
- link to canonical pages,
- and avoid hidden contradictory scoring logic.

JavaScript must not:

- render primary reference content exclusively,
- hide definitions,
- create indexable dynamic result routes,
- use external dependencies by default,
- or make the site dependent on decorative effects.
