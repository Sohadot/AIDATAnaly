## 4. Static-First Technical Posture

This decision ratifies AIDAtanaly's static-first technical posture.

The implementation must prioritize:

- semantic HTML,
- CSS variables,
- minimal JavaScript,
- reference pages readable without JavaScript,
- no external JavaScript dependencies by default,
- no analytics before governance decision,
- no WebGL or 3D unless separately governed,
- no decorative motion,
- performance-safe delivery,
- accessibility-safe design,
- and SEO-safe route structure.

Scanner JavaScript is permitted only as controlled functionality.

Primary reference content must remain available without JavaScript.

## 6. Non-Indexed Sprint Implementation Rule

This decision explicitly ratifies sprint-based implementation under non-indexed status.

AIDAtanaly may be built incrementally across sprints.

However, incremental implementation must remain non-indexed until the full Required Launch architecture is complete.

Allowed non-indexed states include:

- private preview,
- internal review,
- staging environment,
- soft launch with `noindex`,
- or controlled non-indexed demonstration.

No incomplete implementation may be submitted as the official public indexed reference release.

---

## 7. Full 41-Route Public Indexed Release Requirement

This decision ratifies that the first public indexed reference release must include the complete 41-route Required Launch set defined in `ROUTE_MAP.md`.

The first public indexed release must include:

- homepage,
- 3 required category routes,
- 3 required ATI standard routes,
- 4 vector routes,
- TFO overview,
- 22 failure mode and diagnostic constraint routes,
- scanner route,
- ATI Snapshot report route,
- methodology route,
- governance route,
- sources route,
- privacy route,
- terms route.

No incomplete public indexed release may be treated as AIDAtanaly's first official reference launch.

---

## 8. Component Implementation Clarification

This decision ratifies the component implementation rule.

Sprint 2 must implement the core reusable interface foundation.

Any governed component not fully implemented during Sprint 2 may be implemented at the first sprint where it is required by a public page or scanner view.

Delayed component implementation is allowed only if:

- the component remains listed in the governed component registry,
- its first implementation follows `INTERFACE_GOVERNANCE.md`,
- it is reused consistently after introduction,
- and it passes the same interface quality gates as Sprint 2 components.

This preserves practical sprint sequencing without weakening component governance.

## 9. Quality Gate Ratification

Implementation Plan v1.0 ratifies mandatory quality gates before public indexed launch.

The implementation must pass:

- Governance Gate,
- Route Gate,
- Content Gate,
- Interface Gate,
- Scanner Gate,
- Link Gate,
- Performance Gate,
- Accessibility Gate.

Failures must block public indexed release.

No public indexed release may proceed if:

- any Required Launch route is missing,
- internal links are broken,
- JSON data contradicts Markdown governance,
- Evidence Confidence is mathematically included in score,
- E0 produces numeric scores,
- Partial Profile rule is violated,
- failure modes are not TFO-governed,
- WebGL/3D appears without governance,
- or pages contain thin, generic, or unsupported content.

## 10. Public Release Separation

This decision ratifies that `IMPLEMENTATION_PLAN.md` does not authorize public release by itself.

Public launch requires a separate:

```
PUBLIC_RELEASE_PLAN.md
```

The Public Release Plan must confirm:

- route completion,
- quality gate pass,
- sitemap readiness,
- robots policy,
- indexation posture,
- release notes,
- post-launch monitoring,
- rollback procedure,
- and repository governance exposure policy.

Implementation builds the system.

Public Release governs exposure.

---

## 11. Repository Governance Exposure Boundary

This decision does not permanently decide whether repository governance documents are publicly exposed during website publication.

That decision is deferred to `PUBLIC_RELEASE_PLAN.md`.

The Public Release Plan must explicitly choose one of the following policies:

1. exclude repository governance documents from the public website deployment package,
2. expose selected governance documents intentionally as public transparency artifacts,
3. or expose the repository structure as-is with a clear governance transparency rationale.

No accidental exposure policy is permitted.

Repository governance visibility must be intentional.
